# Migrating from v0.x to v1.0

This is the primary upgrade guide for users moving from `0.2.x`, `0.3.x`, or
`0.4.x` to GIS Engine `v1.0.0`.

## Who Should Read This

- SDK consumers using `@gis-engine/engine`
- MCP / orchestration consumers using `@gis-engine/ai`
- CLI users generating or scaffolding projects with `create-gis-map`

## Upgrade Summary

`v1.0.0` does not add a new MCP tool, new runtime command family, or stable
SceneView3D runtime. It does lock the release line around the current SDK + CLI
surface, generated-app evidence workflow, release wording guardrails, and the
published package identities.

## Package Line

Update to:

```json
{
  "dependencies": {
    "@gis-engine/engine": "^1.0.0",
    "@gis-engine/ai": "^1.0.0",
    "@gis-engine/cli": "^1.0.0"
  }
}
```

`@gis-engine/scene3d` remains an experimental package boundary. It does not
promote stable `view.mode: "scene3d"` runtime behavior in `v1.0.0`.

## What Changed Across v0.x

### From 0.2.x

- CLI scaffolding and generation are now part of the stable release line.
- Public wording is stricter about what is blocked: stable SceneView3D,
  runtime-blocked cloud-native loaders, and advanced geoprocessing remain out.
- `maplibre-gl` remains an optional peer dependency when you use the MapLibre
  renderer path.

### From 0.3.x

- Keep the `hashPrompt()` migration from
  [v0.2 to v0.3](./v0.2-to-v0.3.md): prompt hashes use `sha256:<hex>`.
- Real provider HTTP integration remains available, but release wording now
  treats it as part of the supported CLI path instead of an experimental add-on.

### From 0.4.x

- The app template, generated delivery summaries, preflight output, and
  artifact verification flow are now part of the stable developer workflow.
- Generated-app delivery evidence is the supported review surface; do not infer
  runtime support beyond what delivery summaries and wording guardrails allow.

## Behavior And Contract Notes

### MCP surface

- The public MCP tool list remains:
  `validate_spec`, `apply_commands`, `export_spec`, `get_context_summary`,
  `snapshot_spec`, `explain_spec`, `export_example_app`.
- No alias migration is required.

### CLI surface

- No new flags are required for `v1.0.0`.
- Recommended first-run path:

```bash
npm exec --package @gis-engine/cli@latest -- create-gis-map my-map
```

- Recommended generated-app verification path:

```bash
npm exec --package @gis-engine/cli@latest -- create-gis-map --preflight ./map.json --json
npm exec --package @gis-engine/cli@latest -- create-gis-map --verify-artifacts ./my-map --json
```

### Wording and capability expectations

- Scene browsing remains extension-only delivery evidence.
- PMTiles support still means URL-compatible display, readiness, and fixture
  evidence; it does not mean archive parsing, hidden fetches, worker-backed
  runtime queries, or parser promotion.
- GeoParquet, FlatGeobuf, and GeoTIFF remain public source contracts with
  runtime blocked.

## Recommended Upgrade Steps

1. Bump package versions to `^1.0.0`.
2. Re-run your existing validation / snapshot / review pipeline.
3. If you ship generated apps, add `--preflight` and `--verify-artifacts` to
   your CI or reviewer handoff flow.
4. Re-check your docs and product copy against v1.0 wording guardrails so you
   do not imply stable 3D runtime or cloud-native runtime promotion.
