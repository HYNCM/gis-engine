## 差距分析

### P0 — 发布阻断（必须解决）

| # | 问题 | 影响 |
|---|---|---|
| 1 | **无 LICENSE 文件** | 所有 package.json 声明 Apache-2.0 但仓库缺少 LICENSE 文件。外部开发者和企业合规工具会直接拒绝。 |
| 2 | **无 CONTRIBUTING.md** | 外部开发者不知道如何参与贡献、报告 Bug、提交 PR。 |
| 3 | **无 CODE_OF_CONDUCT.md** | GitHub 页面显示"无行为准则"，社区标准缺失。 |
| 4 | **根 README 包含内部内容** | "Project Status"段落（内部 checkpoint/SRC-006/W23）和 AGENTS.md 链接对开发者无意义且造成困惑。 |

### P1 — 发布前必须

| # | 问题 | 影响 |
|---|---|---|
| 5 | **版本号需对齐到 1.0.0** | 当前 0.5.0，需通过 Changesets 统一 bump 到 1.0.0。 |
| 6 | **scene3d 应发布到 `next` tag** | scene3d 仍标记 experimental，不应占 `latest` tag。 |
| 7 | **缺少 Diagnostics 示例** | 结构化诊断是核心差异点，但无独立示例展示。 |
| 8 | **缺少 Snapshot Testing 示例** | 文档站有指南页但无可运行示例。 |
| 9 | **包 README 引用内部文档** | scene3d README 包含 W23/SRC-001 等内部术语，scene3d-three-adapter README 含 5 处 TASK/SRC 引用。engine/ai README 需 Day 2 grep 确认是否干净。 |

### P2 — 发布质量

| # | 问题 | 影响 |
|---|---|---|
| 10 | **无独立 API Reference 生成** | VitePress API 页是手工编写摘要，非 TypeDoc 自动生成，可能遗漏导出。 |
| 11 | **Changesets + publish workflow 集成** | 当前 publish workflow 仍使用 `npm version`，需切换到 changesets action。 |
| 12 | **scene3d-three-adapter 版本策略** | 当前 0.2.0，不在 Changesets linked group 中，不受 GA bump 影响。v1.0 发布时明确保持 0.x experimental 不同步 bump。 |
| 13 | **覆盖率阈值为硬性阻断** | vitest.config.ts 中 thresholds 为 hard threshold，不达标会 CI 失败。需确认当前覆盖率或临时降级为 advisory。 |

## 每日执行计划

### Week 1: 发布基础设施 + 文档打磨

#### Day 1（06-09 周一）—— 法律文件 + 社区标准

- 创建 `LICENSE` 文件（Apache 2.0 全文）
- 创建 `CONTRIBUTING.md`（开发环境搭建、PR 流程、编码规范、Issue 模板）
- 创建 `CODE_OF_CONDUCT.md`（Contributor Covenant v2.1）
- 创建 `.github/ISSUE_TEMPLATE/bug-report.md` 和 `feature-request.md`

#### Day 2（06-10 周二）—— README 清理 + 元数据对齐

- 根 `README.md`：删除 Project Status 段落、移除 AGENTS.md 和内部 docs 链接
- 根 `README.md`：优化 Quick Start 面向外部开发者的体验路径
- `grep` 各包 README 确认无遗漏的内部引用（W23/SRC-/TASK-/checkpoint 等关键词）
- scene3d / scene3d-three-adapter README 清理内部术语
- scene3d README 添加明确的 "Experimental" 声明

#### Day 3（06-11 周三）—— 版本对齐 + Changesets 集成

- 创建 changeset 将 GA 包从 0.5.0 bump 到 1.0.0
- 修改 publish workflow 使用 `@changesets/action` 替代 `npm version`
- 配置 scene3d 发布到 `--tag next`（在 npm-publish.yml 中对 scene3d 包单独加 `--tag next` flag，而非 changeset config 层面处理）
- 确认 scene3d-three-adapter 保持 0.x（不在 linked group 中，不随 GA 包 bump）
- 验证 `pnpm changeset version` 正确生成 CHANGELOG

#### Day 4（06-12 周四）—— 覆盖率验证 + 文档站提前准备

- 运行 `vitest --coverage` 确认当前覆盖率是否达标
- 如未达标，按优先级处理：① 补充关键路径测试（首选）→ ② CI 中拆分为 `vitest run`（无 coverage）+ 独立 coverage job 不阻断 → ③ 临时降低 thresholds 数值
- 提前准备文档站 v1.0 刷新内容（首页 banner、Release Notes 草稿）
- 验证全量构建 + 测试通过

#### Day 5（06-13 周五）—— 文档站 v1.0 刷新

- VitePress 首页添加 v1.0 发布公告 / hero banner
- 更新 API Reference 页面，确保与 1.0.0 导出一致
- 添加 "v1.0 Release Notes" 页面
- 验证 `pnpm docs:build` 构建通过

### Week 2: 示例补全 + 发布验证

#### Day 6（06-16 周一）—— Diagnostics Walkthrough 示例

- 创建 `examples/diagnostics-walkthrough/`
- 展示：schema 错误 → structured diagnostic → SuggestedFix → re-validate
- 包含 README 和可运行代码

#### Day 7（06-17 周二）—— Snapshot Testing 示例

- 创建 `examples/snapshot-testing/`
- 展示：createMap → snapshot → compare → assert
- 配合文档站 snapshot-testing 指南页

#### Day 8（06-18 周三）—— MCP Server Standalone 示例

- 创建 `examples/mcp-server-setup/`
- 展示：MCP server 启动 + Claude Desktop/Cursor 配置 + 工具调用流程
- 包含完整的 `claude_desktop_config.json` 示例

#### Day 9（06-19 周四）—— 发布 Dry Run + 全量回归

- `pnpm publish:dry` 全量 dry-run，验证 tarball 内容
- 全量测试回归：`pnpm check`
- knip 死代码扫描（发现 dead code 记录到 issue，不阻断发布）
- Biome 全量检查
- 验证文档站构建和链接完整性

#### Day 10（06-20 周五）—— 最终审查 + 打标签发布

- 最终 Code Review：确认所有 P0/P1 已解决
- `pnpm changeset version` 生成版本号
- 提交版本变更 + changeset 消费
- `git tag v1.0.0` → 触发 publish workflow
- 验证 npm 包发布成功、文档站部署正常
- 发布 GitHub Release（含 v1.0.0 changelog）

## 风险与缓解

| 风险 | 概率 | 缓解措施 |
|---|---|---|
| Changesets publish workflow 集成出问题 | 中 | Day 3 预留调试时间，Day 9 dry-run 兜底 |
| 覆盖率硬性阈值未达标导致 CI 失败 | 中 | Day 4 提前验证覆盖率；如不达标，补充关键测试或临时降级 thresholds |
| scene3d experimental 定位争议 | 低 | 保持 `--tag next`，README 明确标注 experimental |
| scene3d-three-adapter 版本混淆 | 低 | Changesets ignore 列表已排除；README 标注 experimental 0.x |
