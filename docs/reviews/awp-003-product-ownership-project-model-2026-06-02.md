## Route And Module Boundary

Future product movement, if AWP-007 returns Go, should use the generated-app
review-console product area instead of promoting the example in place:

- Reserved product module: `apps/review-console`.
- Reserved workbench route shape: `/review-console/workbench/:projectId`.
- Current reference implementation: `examples/ai-map-workbench`.
- Current server boundary: local example HTTP server only.

No `apps/review-console` files are created in AWP-003. The route/module boundary
is a reserved target for a future task so that durable audit, review decisions,
authorization, and UI evidence can be designed against one product surface.

## Project Identity Model

The minimum product project identity for future work is:

| Field | Required semantics | Must not contain |
| --- | --- | --- |
| `projectId` | Stable opaque id generated by the server/product layer. | File paths, provider ids, credentials, prompt text, or user-controlled URL fragments. |
| `sessionId` | Ephemeral workbench session id for local review continuity. | Durable auth tokens or raw request bodies. |
| `mapId` | Public map identifier from the active `MapSpec`. | Full `MapSpec` payload. |
| `revision` | Current command-applied map revision. | Command bodies or patches. |
| `auditCursor` | Bounded pointer to payload-free audit evidence. | Raw prompts, provider raw bodies, feature payloads, screenshots, or command payloads. |
| `reviewDecisionIds` | References to future append-only accept/block/follow-up decisions. | Direct map mutation state. |

`AWP-004` durable audit work must scope retention/export to this identity model
before adding storage. `AWP-005` review decisions must attach to
`projectId`/`revision` evidence and remain append-only.
